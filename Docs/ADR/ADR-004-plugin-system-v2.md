# ADR-004: Plugin system v2 (pure declarations)

**Status:** Accepted

## Context

V1 plugins were Go `.so` files loaded at runtime. This approach had fundamental problems:
- Platform-specific (only Linux/macOS — no Windows support)
- Hard to debug (crashes in plugins crash the host process)
- Impossible to validate before loading
- No schema for node configuration parameters
- Required compilation for each platform

## Decision

V2 plugins are **pure declarative JSON manifests** with **zero executable code**:

- `plugin.json` describes nodes, input/output ports, and config schemas (JSON Schema)
- The Generator reads manifests to know what code to generate
- No runtime loading of plugin code
- Auto-discovery from `Plugins/` directory (recursive scan)
- JSON Schema validation of node configurations
- Plugins declare `runtime_bundle` reference for their dependencies

### Manifest Structure

```json
{
  "id": "yolo",
  "name": "YOLO",
  "version": "1.0.0",
  "kind": "algorithm",
  "nodes": [
    {
      "type": "model_trainer.yolo",
      "name": "YOLO Trainer",
      "inputs": [...],
      "outputs": [...],
      "config_schema": { ... }
    }
  ]
}
```

## Consequences

- **Positive:** Safe — plugins can't crash the host
- **Positive:** Cross-platform — manifests are just JSON
- **Positive:** Validatable — schemas are JSON Schema draft-07
- **Positive:** Easy to create — just write a JSON file in Plugins/
- **Positive:** No compilation needed for plugins
- **Negative:** Can't run plugin code directly (must be generated by Generator)
- **Negative:** Static analysis only — no dynamic behavior at plugin load time
- **Negative:** Migration effort from V1 to V2 plugin format
